# 多窗口协作模型

## 1. 角色分工

```
┌─────────────────────────────────────────────────────────────┐
│  主 Agent 对话（本窗口）                                       │
│  struct/ 构想 · command/ 派活 · feedback/ 验收 · 与用户对齐      │
└───────────────────────────┬─────────────────────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
   子 Agent 窗口 A      子 Agent 窗口 B      用户调研
   执行 command/       执行 command/       写入 my_research/
        │                   │
        └─────────┬─────────┘
                  ▼
           feedback/<TASK_ID>.md
                  │
                  ▼
           用户带回主 Agent 验收
```

| 角色 | 做什么 | 不做什么 |
|------|--------|----------|
| **主 Agent** | 目标、边界、契约、Done Definition、派 `command/` | 不写逐步实现说明、不指定类名/函数签名 |
| **子 Agent** | 在约束内**自行设计并实现** | 不擅自扩大 scope、不违反契约 |
| **用户** | 调研、在窗口间传递、最终决策 | — |

---

## 2. 子 Agent 边界规则

1. **执行者，非决策者** — 四决策、MVP 范围以 `struct/` 为准；有异议写入 `feedback/`，不静默偏离
2. **未批准不写实现** — 任务单 `可以写代码: 否` 时禁止改 `mini_coding_agent.py` 业务逻辑
3. **最小改动** — 只动任务单允许的文件
4. **必须可验证** — 实现类任务附 pytest；调研类任务附结论与引用
5. **回报落盘** — 完成写入 `feedback/<TASK_ID>.md`，不只在聊天里说完
6. **保留用户注释** — 除非该段代码逻辑已完全删除，否则不要删除或「顺手清理」注释；拆文件/迁函数时注释随代码迁移
7. **新增代码带注释** — 实现时为新模块、公共 API、非显而易见分支与契约写**适量**注释，保证可读性；避免无信息注释与逐行复述代码

## 3. 用户操作步骤

### 派发子 Agent

在新 Cursor 对话中：

```
你是本子项目的子 Agent。
请先阅读：
- @docs/struct/README.md
- @docs/command/<TASK_ID>.md

严格执行任务单，完成后将回报写入 docs/feedback/<TASK_ID>.md
（格式见 docs/feedback/TEMPLATE.md）
```

### 带回主 Agent

将 `feedback/<TASK_ID>.md` 内容或路径发回主 Agent 对话，由主 Agent 验收并更新 `struct/README.md` 状态板。

## 4. 常见问题

**Q: 子 Agent 能否自己做战略决定？**  
A: 不能。建议写入 feedback，由主 Agent 与用户裁决。

**Q: 调研和实现能否同一窗口？**  
A: 可以，但须分开任务单；实现任务必须 `可以写代码: 是`。

**Q: 何时拆 `mini_coding_agent.py`？**  
A: Phase 1 单文件扩展；拆模块需 struct 中明确设计且主 Agent 批准。

**Q: 用户调研放哪？**  
A: `docs/my_research/`，不要与子 Agent 的 `feedback/` 混用。

## 方案摘要

### 模块布局

```
mini_coding_agent/harness/
  __init__.py       # 导出 INTENT_IDS, GateResult, classify_gate, handle_ask
  types.py          # INTENT_IDS（5 类）、GateResult、GATE_MAX_NEW_TOKENS
  gate_prompt.py    # build_gate_prompt：五类定义 + 边界示例 + JSON schema
  gate.py           # classify_gate / parse_gate_response（1× complete）
  runner.py         # handle_ask：Gate → 持久化 last_gate → open 降级
```

### 关键流程

```
用户消息
  → handle_ask(agent, message, harness_enabled, gate_log)
      ├─ harness off 且 gate_log off → agent.ask()（无额外 LLM）
      └─ 否则 → classify_gate（1× complete + gate prompt）
            → session["last_gate"] = GateResult.to_session_dict()
            → 可选 stderr [gate] 行（--gate-log）
            → route=harness_pipeline 且 harness on → stderr 提示降级
            → agent.ask()（5.1 始终 open 执行）
```

### Gate 解析规则

| 情况 | confidence | route |
|------|------------|-------|
| 合法 JSON + intent∈五类 + high | high | harness_pipeline |
| 合法 JSON + intent∈五类 + low | low | open |
| JSON 解析失败 | low | open |
| intent 不在 INTENT_IDS | low | open |

**规则兜底仅上述**；不做规则替代 LLM 主分类。

### CLI 接线

- `--harness off|on`（默认 **off**）
- `--gate-log`：stderr 打印 `[gate] intent_id=… confidence=… route=… skill=…`
- REPL 与 one-shot 均经 `handle_ask()`，保留 `agent.ask()` 为唯一执行路径

### Session 字段

- `session["last_gate"]`：`{intent_id, confidence, route, skill?}`，每次 Gate 后落盘

### 明确未做（留 5.2+）

- `templates/`、`planner`、`executor`、`pipeline`、`rig/`、节点实现
